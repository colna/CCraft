# DevChat — 任务计划

> 来源：`docs/01-产品设计文档.md`、`docs/02-交互设计文档.md`、`docs/03-技术架构文档.md`。本文件已融合原产品任务计划与真实实现计划，是后续唯一任务计划入口。

## 1. 真实完成定义

一个任务只有同时满足以下条件才算真实完成：

- 运行时使用真实服务、真实本地持久化或真实设备能力；mock 只允许出现在测试、fixture 或本地 preview 的明确降级路径中。
- 前端不保存、不读取 API Key 或 GitHub Token 明文，只传 `secretRef` 或授权状态。
- Rust/Tauri command 边界负责读取 secret、校验外部输入、调用真实 service、返回结构化错误。
- 关键行为有自动化测试；外部 API 通过本地 fake server 或 fixture 测试，不触碰真实凭据。
- 完成后运行对应验证命令，并追加 `docs/任务报告.md`。

## 3. 命令与质量门禁

- App test: `pnpm --filter devchat test`
- App lint: `pnpm --filter devchat lint`
- App build: `pnpm --filter devchat build`
- Rust tests: `cd apps/devchat/src-tauri && cargo test`
- Web test: `pnpm --filter web test`
- Web lint: `pnpm --filter web lint`
- Web build: `pnpm --filter web build`
- Root gate: `pnpm test && pnpm lint && pnpm build`

每个任务至少运行直接相关命令；每个 checkpoint 运行 root gate 和 Rust tests。

## 17. 风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| 平台级 secret 插件移动端能力不足 | 高 | 先封装 `SecretStore` trait，桌面/移动分别适配，缺口写入 ADR |
| GitHub OAuth Proxy 未确定 | 高 | R1.3 单独决策，不阻塞 PAT 过渡测试，但发布前必须完成 |
| GitHub Git Data API 与 patch apply 复杂 | 高 | R4.2 和 R5.1 分开，先纯函数测试 patch，再接 API |
| AI 输出 diff 不稳定 | 中 | Prompt 输出合同 + parser 容错 + 无法解析时不生成假变更 |
| 大仓库 snapshot 慢或触发 rate limit | 中 | 限制文件大小、缓存、进度事件、错误映射 |
| 移动端 deep link/store/http 与桌面差异 | 中 | 每个涉及平台能力的 checkpoint 加模拟器验证记录 |
| 官网下载入口早于真实包发布 | 中 | 明确平台状态，不放误导性下载按钮 |
