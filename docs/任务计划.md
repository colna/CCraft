# DevChat — 任务计划

> 来源：`docs/01-产品设计文档.md`、`docs/02-交互设计文档.md`、`docs/03-技术架构文档.md`

## 1. 计划目标

本计划将 DevChat 拆解为可实施、可验证的任务。项目采用 Turborepo monorepo，包含主项目 App（`apps/devchat`，Tauri 2.0 + React + Vite + Tailwind）和官网 App（`apps/web`，Next.js App Router + Tailwind），共享能力沉淀到 `packages/*`。

## 2. 执行原则

- 每个任务完成后必须保证仓库处于可构建、可测试、可继续开发的状态。
- 主项目 App 的敏感数据只能在 Rust/Tauri 侧处理，WebView 前端不接触 API Key 和 GitHub Token 明文。
- 官网 App 只承载公开内容、下载入口和文档入口，不处理用户密钥、项目快照或 GitHub Token。
- UI/UX 实现时先读取本仓库 UI/Next skills：`apple-design`、`frontend-design`、`vercel-react-best-practices`、`next-best-practices`。
- 涉及逻辑或行为变更时补测试；涉及安全边界时同步使用 `security-and-hardening`。

## 3. 依赖关系概览

```
Monorepo 基础
  ├── 共享配置与类型
  │   ├── 主项目 App 壳层
  │   │   ├── Rust 安全存储
  │   │   ├── GitHub OAuth / Repo API
  │   │   ├── 项目快照
  │   │   ├── AI 流式对话
  │   │   ├── Diff 解析与审查
  │   │   └── Commit & Push
  │   └── 官网 App 壳层
  │       ├── 首页 / 功能 / 下载 / 文档
  │       └── SEO / 静态内容
  └── 测试、lint、构建流水线
```

## 4. Phase 1 — MVP

目标：完成可端到端演示的主项目 App 与官网 App。用户可以配置 AI、绑定 GitHub、选择仓库、生成项目快照、通过对话获得 Diff、审查并提交；官网可以完成产品介绍、下载入口和文档入口。

**描述：** 建立 `pnpm` + `turbo` monorepo 基础结构，为 `apps/devchat`、`apps/web` 和 `packages/*` 提供统一脚本。

**验收标准：**
- [ ] 根目录包含 `package.json`、`pnpm-workspace.yaml`、`turbo.json`。
- [ ] 根命令 `pnpm dev`、`pnpm build`、`pnpm test`、`pnpm lint` 可被 Turborepo 调度。
- [ ] 生成产物目录已加入忽略规则。

**验证：**
- [ ] `pnpm install`
- [ ] `pnpm build`

## 7. 风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| Tauri 移动端插件能力与桌面差异 | 高 | 每个涉及 deep link、store、http、widget 的任务都在 iOS/Android 模拟器验证 |
| GitHub OAuth 需要代理交换 secret | 高 | Phase 1 提前完成 OAuth Proxy 设计，避免把 client secret 放入 App |
| AI 返回 diff 格式不稳定 | 中 | 设计 diff parser 的容错和错误反馈，必要时约束 system prompt 输出格式 |
| GitHub Git Data API 提交流程复杂 | 中 | 使用 mock 覆盖六步流程，并在测试仓库手动验证 |
| 官网和主 App 视觉分叉 | 中 | 共享 Tailwind token 和 UI 原语，重要视觉变更同时检查两个 app |
| 任务范围膨胀 | 中 | 每个任务保持 1-5 个主要文件范围，超过则拆分新任务 |
